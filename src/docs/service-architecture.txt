Pixel Quest - Service-Architektur Dokumentation

1. ÜBERSICHT
============
Das State-Management des Pixel Quest Games basiert auf einer modularen Architektur mit Angular Signals.
Es gibt drei Haupt-Services, die jeweils für einen Teil des Spielzustands zuständig sind:
- WalletService (Geldbörse)
- SkillsService (Fertigkeiten)
- ProfileService (Spielerprofil)
Zusätzlich gibt es den GameStateService als zentrales Koordinationsmittel.

2. GAMESTATE-SERVICE (ZENTRAL)
===============================
Der GameStateService ist der zentrale Koordinator aller Sub-Services.

2.1. Hauptfunktionen:
- Verwaltet die aktuelle Character-ID als Angular Signal
- Steuert den Login-Prozess für Charaktere
- Läd alle Daten für einen Charakter aus localStorage
- Speichert Daten in localStorage
- Stellt Default-Daten für neue Charaktere bereit

2.2. Methoden:
- loginCharacter(charId): Setzt Character-ID und lädt alle Daten
- crawlData<T>(dataType, charId): Lädt Daten aus localStorage
- saveData(dataType, data): Speichert Daten in localStorage
- getDefaultData(dataType): Gibt Standardwerte für neue Charaktere zurück

3. WALLETSERVICE (GELDBÖRSE)
============================
Verwaltet das Gold und Rubies des Spielers.

3.1. Interne Struktur:
- Internes Signal: { gold: number, rubies: number }
- Readonly Signals für Komponenten: gold, rubies

3.2. Methoden:
- init(data): Initialisiert mit Daten
- addGold(amount): Fügt Gold hinzu und speichert
- spendGold(amount): Zieht Gold ab, wenn genug vorhanden

4. SKILLSSERVICE (FERTIGKEITEN)
===============================
Verwaltet die Fertigkeiten des Spielers.

4.1. Interne Struktur:
- Internes Signal: { attack: number, defense: number, spells: string[] }
- Readonly Signals für Komponenten: attack, defense, spells

4.2. Methoden:
- init(data): Initialisiert mit Daten
- upgradeAttack(): Erhöht Angriffswert
- upgradeDefense(): Erhöht Verteidigungswert
- addSpell(spell): Fügt Zauber hinzu

5. PROFILESERVICE (SPIELERPROFIL)
=================================
Verwaltet den Spielername und das Level.

5.1. Interne Struktur:
- Internes Signal: { name: string, level: number, exp: number }
- Readonly Signals für Komponenten: name, level, exp

5.2. Methoden:
- init(data): Initialisiert mit Daten
- updateName(name): Ändert Spielernamen
- addExp(amount): Fügt Erfahrungspunkte hinzu
- levelUp(): Erhöht Level (setzt Erfahrung zurück)

6. DATENFLUSS ZWISCHEN DEN SERVICES
==================================
6.1. Login-Prozess:
1. loginCharacter() wird aufgerufen
2. GameStateService lädt alle Daten für den Charakter
3. Für jeden Datentyp wird crawlData() aufgerufen
4. Jeder Service erhält seine Daten via init() Methode

6.2. Datenänderung:
1. Komponente ruft Methode des Services auf (z.B. addGold())
2. Service ändert internen Zustand
3. Service ruft saveData() des GameStateServices auf
4. GameStateService speichert Daten in localStorage

6.3. Datenstruktur:
- Jeder Service speichert Daten unter eigenem Key: [charId]_[serviceName]
- Beispiel: char123_wallet, char123_skills, char123_profile

7. BEISPIEL-ANWENDUNG
====================
1. Spieler meldet sich an:
   - GameStateService.loginCharacter("char456")

2. Spiel lädt Daten:
   - GameStateService.crawlData("wallet", "char456") 
   - GameStateService.crawlData("skills", "char456")
   - GameStateService.crawlData("profile", "char456")

3. Services initialisieren sich:
   - WalletService.init({gold: 100, rubies: 5})
   - SkillsService.init({attack: 10, defense: 8, spells: ["Feuerball"]})
   - ProfileService.init({name: "Spieler1", level: 5, exp: 120})

4. Spieler fügt Gold hinzu:
   - WalletService.addGold(50)
   - Interner Zustand wird geändert
   - GameStateService.saveData("wallet", {gold: 150, rubies: 5})
   - Daten werden im localStorage gespeichert
